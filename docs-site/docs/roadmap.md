# 로드맵

:::caution 현재 상태
WorkTwin은 **아직 구현되지 않은 아이디어 단계**입니다. 아래 로드맵은 확정된 개발 일정이 아니라, 개념을 실현하기 위해 필요한 단계를 논리적 순서로 정리한 것입니다.
:::

## 단계별 비전

### 1단계 — 읽고 검색하는 WorkTwin

받은 메일함 동기화, 로컬 DB, RAG 벡터 DB 색인, 챗봇을 통한 의미 기반 검색까지가 최소 성립 범위입니다. 이 단계만으로도 "수 기가바이트 메일함을 대화로 탐색한다"는 핵심 가치는 성립합니다. → [메일함 동기화·RAG 색인](./features/mailbox-sync-rag.md), [메일함 챗봇 검색](./features/chatbot-inbox-search.md)

### 2단계 — 나를 분석하는 WorkTwin

보낸 메일함을 분석해 답장/리뷰 패턴과 스타일을 추출합니다. 이 시점에는 아직 자동화하지 않고, "나는 이런 기준으로 판단해왔다"는 것을 사용자에게 보여주는 것 자체가 가치가 됩니다. → [보낸 메일함 페르소나 분석](./features/sent-mail-persona-analysis.md)

### 3단계 — 대신 답하는 WorkTwin

반복적이고 어렵지 않은 답장 패턴을 agent skill로 만들어, 초안 제안부터 시작해 점진적으로 자동 발송까지 나아갑니다. → [자동 답장 스킬](./features/auto-reply-skill.md)

### 4단계 — 역할로 나뉘는 WorkTwin

메일함 전체 분석을 통해 역할을 추출하고, 역할별 서브에이전트로 승격시킵니다. 이 단계에서 WorkTwin은 단일 에이전트가 아니라 **역할별로 특화된 에이전트의 집합**이 됩니다. → [역할 추출과 서브에이전트](./features/role-extraction-subagents.md)

## 아직 열려 있는 질문

각 단계를 실제로 구현하기 전에 답이 필요한 질문들입니다.

- **"어렵지 않은 일"의 기준을 어떻게 정량화할 것인가?** (반복 빈도, 판단 분기의 단순성, 리스크 수준 등을 조합해야 할 것으로 보이나 구체적인 기준은 미정)
- **역할 간 경계는 어떻게 나눌 것인가?** 한 사람의 역할은 명확히 분리되기보다 겹치는 경우가 많을 수 있습니다.
- **로컬 DB / RAG 벡터 DB의 구체적인 기술 스택은?** 수십 기가바이트 규모에서 실용적인 임베딩·인덱싱 방식에 대한 검증이 필요합니다.
- **자동 발송의 신뢰 기준과 되돌리기 장치는?** ([로컬 우선·프라이버시](./privacy-local-first.md) 참고)
- **제3자 정보와 삭제/철회 가능성을 어떻게 보장할 것인가?** ([로컬 우선·프라이버시](./privacy-local-first.md) 참고)

이 질문들에 대한 답이 쌓일수록 이 로드맵 문서도 함께 갱신되어야 합니다.
