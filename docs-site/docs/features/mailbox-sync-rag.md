# 메일함 동기화 · RAG 색인

**동기화(Sync) 탭**은 WorkTwin의 모든 기능이 시작되는 지점입니다. 메일 계정을 연결하면, 받은 메일함(그리고 필요 시 보낸 메일함)을 로컬 DB로 가져와 분류·정리한 뒤 RAG 벡터 DB에 색인합니다.

## 왜 그냥 검색이 아니라 RAG인가

메일 클라이언트의 기본 검색은 제목/발신자/키워드 매칭 수준입니다. 반면 실제로 필요한 질문은 다음과 같은 형태입니다.

- "작년에 이 프로젝트 관련해서 오갔던 논의 흐름을 요약해줘"
- "이 사람과 나눈 메일 중에 일정이 바뀐 부분만 뽑아줘"
- "이 주제에 대해 내가 이전에 뭐라고 답했는지 찾아줘, 정확한 단어는 기억 안 나"

이런 질문에 답하려면 키워드가 아니라 **의미 기반 검색**이 필요하고, 검색된 메일을 근거로 답을 조합해주는 **생성** 단계가 필요합니다. 이 두 가지를 합친 것이 RAG(Retrieval-Augmented Generation)입니다.

## 처리 단계

1. **수집**: 메일 계정에서 메일 원문과 메타데이터(발신자/수신자, 스레드 ID, 타임스탬프, 라벨/폴더)를 가져와 로컬 DB에 저장합니다.
2. **분류 및 주제별 정리**: 프로젝트, 발신자 그룹, 스레드 등을 기준으로 메일을 주제 단위로 묶습니다. 이 정리는 이후 [역할 추출](./role-extraction-subagents.md)의 입력 데이터로도 재사용됩니다.
3. **색인**: 정리된 메일 본문을 임베딩해 RAG 벡터 DB에 저장합니다. 첨부파일 텍스트, 스레드 문맥 등도 색인 대상에 포함할 수 있습니다.
4. **증분 동기화**: 최초 동기화 이후에는 신규/변경된 메일만 반영합니다. 수 기가바이트 규모의 메일함을 매번 통째로 재처리하는 것은 비현실적이기 때문에, 동기화 시점 이후의 변경분만 추적하는 방식이 기본 전제입니다.

## 로컬 DB와 벡터 DB의 역할 분리

| | 로컬 DB | RAG 벡터 DB |
|---|---|---|
| 저장 대상 | 메일 원문 + 메타데이터 (source of truth) | 임베딩된 본문 (검색 전용 인덱스) |
| 용도 | 메일함 탭 탐색, 원본 열람, 재색인의 근거 | 챗봇 탭의 의미 기반 검색 |
| 갱신 방식 | 증분 수집 | 로컬 DB 변경에 맞춰 재색인 |

이렇게 역할을 나누면 벡터 DB의 임베딩 모델이나 인덱스 구조를 나중에 교체하더라도, 로컬 DB의 원본 데이터만 있으면 재색인이 가능합니다.

동기화된 데이터는 [메일함 챗봇 검색](./chatbot-inbox-search.md)에서 실제로 어떻게 활용되는지 이어서 확인할 수 있습니다.
