# 메일함 챗봇 검색

**메일함(Inbox) 탭**과 **챗봇(Chatbot) 탭**은 [동기화 탭에서 만든 로컬 DB·RAG 벡터 DB](./mailbox-sync-rag.md)를 사용자가 실제로 소비하는 접점입니다.

## 메일함 탭: 구조화된 탐색

메일함 탭은 동기화 시 만들어진 분류·주제별 정리를 바탕으로 메일을 탐색하는 화면입니다. 기본 메일 클라이언트처럼 폴더/라벨을 오가는 대신, **주제 단위로 묶인 메일 뭉치**를 단위로 훑어볼 수 있습니다.

## 챗봇 탭: 대화형 질의응답

챗봇 탭은 RAG 벡터 DB를 근거로 자연어 질문에 답하는 인터페이스입니다. 전형적인 사용 시나리오는 다음과 같습니다.

- **탐색형 질문**: "이 프로젝트에서 지금까지 나온 주요 이슈들을 시간순으로 정리해줘"
- **회상형 질문**: "작년 이맘때쯤 A사와 나눴던 계약 조건 관련 메일 찾아줘"
- **비교형 질문**: "이 두 벤더 제안서에 대해 내가 각각 뭐라고 답했는지 비교해줘"
- **역할 특정 질문**: "리뷰어로서 이런 종류의 문제에 보통 어떤 기준으로 피드백했지?" (→ [역할별 서브에이전트](./role-extraction-subagents.md) 컨텍스트 활용)

챗봇은 질문에 답할 때 **근거가 된 원본 메일을 함께 제시**하는 것을 기본 원칙으로 합니다. RAG 구조 특성상 답변은 항상 검색된 실제 메일에 기반해야 하고, 사용자가 그 근거를 바로 확인할 수 있어야 신뢰할 수 있는 도구가 됩니다.

## 왜 "수 기가바이트도 문제없다"인가

일반적인 검색 UX는 데이터 규모가 커질수록 사용성이 떨어집니다(스크롤, 필터 조합, 정확한 키워드 기억 필요). 반면 RAG 기반 챗봇은 의미 기반 검색이므로, 데이터가 많을수록 오히려 "이런 게 있었을 텐데" 수준의 모호한 질문에도 답을 찾아낼 가능성이 높아집니다. 즉 메일함 규모는 이 접근에서 약점이 아니라 **자산의 크기**가 됩니다.

이 두 탭에서 소비되는 데이터가 어떻게 신뢰할 수 있는 방식으로 로컬에 보관되는지는 [로컬 우선·프라이버시](../privacy-local-first.md)에서 다룹니다.
