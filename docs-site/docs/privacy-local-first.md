# 로컬 우선 · 프라이버시

이메일함은 업무 데이터 중에서도 가장 민감한 축에 속합니다. 계약 조건, 인사 관련 논의, 미공개 의사결정, 제3자의 개인정보가 모두 메일 안에 섞여 있습니다. WorkTwin이 다루는 데이터의 성격상, 프라이버시 원칙은 부가 기능이 아니라 **아키텍처 결정의 출발점**이어야 합니다.

## 기본 원칙: 로컬 우선

- **동기화, 분류, RAG 색인, 페르소나 분석 — 이 모든 처리는 원칙적으로 사용자의 로컬 환경에서 이루어집니다.** [Tauri 기반 데스크톱 앱](./architecture.md)을 선택한 이유도 여기에 있습니다.
- 로컬 DB와 RAG 벡터 DB는 사용자의 기기에만 저장되며, 임의로 외부로 전송되지 않는 것을 기본값으로 합니다.
- 만약 답변 생성 등 일부 단계에서 외부 LLM API 호출이 필요하다면, **무엇이 전송되는지(원문 전체인지, 검색된 일부 발췌인지)를 사용자가 명확히 인지하고 통제**할 수 있어야 합니다.

## 아직 열려 있는 신뢰 질문

이 프로젝트는 아직 구현 전 단계이기 때문에, 다음과 같은 질문들은 의도적으로 미해결 상태로 남겨둡니다. 실제 설계 단계에서 반드시 다시 짚어야 할 지점입니다.

1. **완전 로컬 LLM vs. 외부 API 혼용**: RAG 검색은 로컬에서 처리하더라도, 답변 생성에 성능이 검증된 외부 LLM을 쓸지, 로컬 모델만으로 충분할지는 트레이드오프가 있습니다.
2. **자동 답장의 발송 권한 범위**: [자동 답장 스킬](./features/auto-reply-skill.md)이 사람의 확인 없이 실제로 메일을 발송하는 것을 어느 수준까지 허용할지는 프라이버시보다는 신뢰·안전 문제에 가깝지만, 같은 맥락에서 함께 설계되어야 합니다.
3. **제3자 정보 처리**: 내 메일함에는 나뿐 아니라 나와 메일을 주고받은 제3자의 정보도 포함됩니다. 이들의 정보를 학습/색인 대상에 포함하는 것에 대한 최소한의 원칙(예: 조직 내부 커뮤니케이션으로 한정 등)이 필요합니다.
4. **삭제·철회 가능성**: 특정 메일이나 스레드를 로컬 DB/벡터 DB에서 완전히 제거하고 싶을 때, 이를 보장할 수 있는 장치가 있어야 합니다.

## 지금 시점에서의 입장

이 문서 작성 시점에서 WorkTwin은 구현되지 않은 아이디어 단계이므로, 위 질문들에 대한 최종 답은 아직 없습니다. 다만 **"로컬 우선을 기본값으로 하고, 예외를 만들 때는 사용자가 명시적으로 인지하고 동의한다"**는 방향성만은 이 프로젝트의 전제로 고정해두고자 합니다. 나머지 열린 질문들은 [로드맵](./roadmap.md)에서 계속 추적합니다.
