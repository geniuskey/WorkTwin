WORKTWIN · 개인 업무 에이전트

내 메일함으로 만드는,
나보다 나를 잘 아는 에이전트

수년간 쌓인 메일함은 내가 어떤 일을 어떻게 해왔는지에 대한 가장 촘촘한 기록입니다.

인사 후 한 문장: "이미 갖고 있는 데이터로 나를 대신할 에이전트를 만드는 이야기를 하겠습니다."
WHY

"그 메일,
다시 찾을 수 있나요?"

3년 전 그 프로젝트에서, 나는 그 이슈에 뭐라고 답했을까요?

여기서 실제로 손을 들어보게 한다 — "최근에 옛날 메일 찾다가 포기하신 분?" 청중 반응을 받고 넘어간다.
WHY

답은 분명히 있는데, 못 찾는다

🔍 키워드가 기억 안 남

정확한 단어를 몰라 검색 자체가 실패합니다.

📚 결과가 너무 많음

수백 건이 나오면 없는 것과 같습니다.

🧩 맥락이 흩어짐

스레드가 나뉘어 흐름을 재구성할 수 없습니다.

검색은 되는데 찾아지지가 않는다 — 이게 진짜 문제입니다.

"검색이 안 되는 게 아니라 찾아지지 않는다"는 표현을 강조. 키워드 매칭의 한계를 짚는다.
PROBLEM

메일함은 이미 나의 업무 기록이다

🎭 역할

어떤 조직에서 어떤 역할을 맡아왔는지.

⚖️ 판단

어떤 기준으로 승인·반려·보류했는지.

🔁 패턴

반복되는 정형 커뮤니케이션의 문법.

사람의 기억은 미화되지만, 메일은 그 순간 실제로 무엇을 했는지를 그대로 남깁니다.

이력서나 개인 위키보다 메일함이 더 정직하고 촘촘하다는 점을 강조.
PROBLEM

그런데 이 자산은 비활성 상태다

😫 검색은 되는데 대화는 안 된다

"내가 예전에 뭐라고 답했더라?"에 답해주지 못합니다.

🔁 같은 판단을 매번 반복한다

패턴이 뻔한 답장도 사람이 매번 새로 씁니다.

🤷 내 역할을 나도 모른다

정리되지 않으면 위임도 자동화도 불가능합니다.

🔒 클라우드에 올리기는 꺼려진다

업무 메일은 외부 SaaS에 통째로 맡기기 어렵습니다.

네 가지 페인포인트를 빠르게 짚고 넘어간다. 마지막 항목은 뒤의 프라이버시 슬라이드로 이어지는 복선.
CONCEPT

WorkTwin

내 메일함 전체를 학습해, 나를 대신해 메일 맥락을 검색·이해·응답하는 개인 전용 에이전트

처음부터 완성된 게 아니라, 메일함을 통해 점진적으로 나를 학습해가는 존재입니다.

이름 그대로 "업무상의 나(Work)"를 복제한 "쌍둥이(Twin)". 한 문장 정의를 또박또박 읽어준다.
CONCEPT

세 가지 축으로 나를 복제한다

📥아는 것받은 메일함 → RAG
▶
✍️답하는 것보낸 메일함 → 답장 스킬
▶
🎭맡은 것메일함 전체 → 역할

각각 다른 데이터에서 다른 능력이 나옵니다.

손가락 세 개를 세우듯 명확히 구분해서 말한다. 이후 세 장이 각 축의 상세.
축 1 · 아는 것

받은 메일함 → RAG 벡터 DB

키워드 매칭으로는 답할 수 없는 질문에 답하기 위해, 메일을 의미 기반으로 색인합니다.

의미 기반 검색 근거 메일 제시
챗봇 질문 예시
작년에 이 프로젝트에서 오갔던
논의 흐름을 시간순으로 요약해줘

메일함이 클수록 유리해집니다 — 규모는 약점이 아니라 자산의 크기입니다.

RAG를 모르는 청중이 있으면 "검색 + 그 결과를 근거로 답 생성"이라고 한 줄로 풀어준다.
축 2 · 답하는 것

보낸 메일함 → 답장 Agent Skill

보낸 메일 한 통 한 통은 텍스트가 아니라 판단의 흔적입니다.

📊 패턴 추출

어떤 요청에 어떻게 답했는지 유형별로 묶습니다.

🎚️ 난이도 판별

반복성과 판단 난이도로 자동화 대상을 가립니다.

⚡ 스킬화

쉬운 유형은 초안 작성 → 자동 발송으로.

동기화가 계속될수록 커버리지가 넓어지는 성장형 기능입니다.

"모든 답장을 자동화하자는 게 아니다"를 반드시 짚는다. 어려운 판단은 사람 몫으로 남긴다.
축 3 · 맡은 것

메일함 전체 → 역할별 Sub-agent

한 사람이 동시에 수행하는 여러 역할이 메일 패턴에서 드러납니다.

프로젝트 리드 리뷰어 벤더 커뮤니케이션 멘토 
# 역할 → 업무 → 액션
역할: 프로젝트 리드
  업무: 일정 변경 승인
    - 일정 영향도 확인
    - 승인/반려 답장

부산물: 스스로도 정리해본 적 없는 내 역할 목록이 만들어집니다.

역할별로 판단 기준·어조·검색 범위가 달라진다는 점이 핵심. 부산물 이야기에서 청중이 가장 크게 반응하는 편.
SCENARIO · 가상

완성됐다면, 이런 하루

1아침 · 회상"B사 계약 조건 변경, 예전에 어떻게 답했지?" → 2년 전 사례를 근거와 함께
2오전 · 답장정형 일정 조율 메일 5통 → 초안이 이미 준비되어 있고, 확인 후 일괄 발송
3오후 · 점검설정 탭에 정리된 내 역할 목록이 각각 서브에이전트로 나열

주의 아직 구현 전 단계이며, 개념 전달을 위한 가상 시나리오입니다.

반드시 "이건 가상 시나리오"라고 명시하고 넘어간다. 과장했다는 인상을 주면 신뢰를 잃는다.

🔄 동기화 · 📬 메일함

메일 수집 → 로컬 DB → 분류·정리 → RAG 색인. 정리된 결과를 주제 단위로 탐색합니다.

💬 챗봇 · ⚙️ 설정

메일함 전체에 자연어로 질의. 역할·스킬·자동 답장 허용 범위를 관리합니다.

웹 서비스가 아니라 데스크톱 앱을 고른 이유가 다음 슬라이드(프라이버시)로 이어진다고 예고한다.
ARCHITECTURE

수집 → 분류 → 색인 → 활용

1수집메일 원문과 메타데이터를 로컬 DB에 저장 — source of truth
2분류·정리프로젝트·발신자·스레드 기준으로 주제 단위 묶음 생성
3색인본문을 임베딩해 RAG 벡터 DB에 저장 — 검색 전용 인덱스
4증분 동기화이후에는 변경분만 반영 — 수십 GB 재처리는 비현실적

원본과 인덱스를 분리하면, 임베딩 모델을 바꿔도 재색인만 하면 됩니다.

역할 분리(로컬 DB = 원본, 벡터 DB = 인덱스)가 왜 중요한지 한 문장으로 설명하고 넘어간다.
PRIVACY

그래서 로컬 우선이 기본값이다

메일함은 계약 조건·인사 논의·미공개 의사결정이 뒤섞인 가장 민감한 업무 데이터입니다.

✅ 원칙

동기화·분석·색인은 사용자 기기에서. 저장물은 임의로 외부에 나가지 않습니다.

⚠️ 예외를 둘 때

무엇이 전송되는지 사용자가 명확히 인지하고 통제할 수 있어야 합니다.

프라이버시는 부가 기능이 아니라 아키텍처 결정의 출발점이었다는 점을 강조. 데스크톱 앱을 고른 이유와 연결.
ROADMAP

네 단계로 점진적으로

1️⃣읽고 검색동기화 + RAG 챗봇
▶
2️⃣나를 분석페르소나 추출
▶
3️⃣대신 답변답장 스킬
▶
4️⃣역할 분화서브에이전트

1단계만으로도 핵심 가치는 이미 성립합니다.

1단계가 MVP라는 점을 명확히. 4단계까지 다 만들어야 쓸모 있는 게 아니라는 메시지.
